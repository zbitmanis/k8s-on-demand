== Overview

This document is the operator reference for secrets management. For the concise rule set used by Claude Code, see link:secrets.md[secrets.md].

Secrets are stored in **AWS Secrets Manager** (never in Git). **External Secrets Operator** syncs them into Kubernetes Secrets. Each workload accesses only secrets in its own path prefix via **IRSA role conditions**.

== Secret Storage Architecture

=== Naming Convention

All secrets follow a path-based hierarchy:

[source]
----
/<tenant-id>/<component>/<secret-name>
/platform/<component>/<secret-name>

Examples:
  /acme-corp/database/postgres-password
  /acme-corp/app/api-key
  /globex/registry/pull-secret
  /platform/webhook-hmac-secret
  /platform/github-token
  /platform/slack-webhook
----

=== AWS Secrets Manager Configuration

Each secret is stored as **JSON**. Format depends on consumer:

**Platform webhooks (Argo Events):**
[source,json]
----
{
  "value": "hex-encoded-hmac-value"
}
----

**GitHub token:**
[source,json]
----
{
  "token": "ghp_..."
}
----

**Slack webhook:**
[source,json]
----
{
  "url": "https://hooks.slack.com/services/T.../B.../X..."
}
----

**Database password:**
[source,json]
----
{
  "password": "the-password",
  "username": "db-user"
}
----

=== Scope & Access Control

[cols="2,3,3"]
|===
| Scope | Path | Access Role

| Platform
| `/platform/*`
| `platform-argo-workflow-runner` (Kubernetes group: `platform:workflow-runner`)

| Tenant
| `/<tenant-id>/*`
| `<tenant-id>-external-secrets` IRSA role (in tenant namespace)
|===

**IRSA policy example (tenant):**

[source,json]
----
{
  "Effect": "Allow",
  "Action": [
    "secretsmanager:GetSecretValue",
    "secretsmanager:DescribeSecret"
  ],
  "Resource": "arn:aws:secretsmanager:<region>:<account-id>:secret:/<tenant-id>/*"
}
----

== Platform Secrets (Managed by Platform Team)

=== Webhook HMAC Secret

**Path:** `/platform/webhook-hmac-secret`

**Format:**
[source,json]
----
{
  "value": "<64-char-hex-string>"
}
----

**Usage:** HMAC-SHA256 signature validation for Argo Events webhook endpoints:
- `/onboard-tenant`
- `/offboard-tenant`
- `/suspend-cluster`
- `/resume-cluster`

**Generation:**
[source,bash]
----
openssl rand -hex 32
----

**Rotation:** Manual. Update the secret in AWS Secrets Manager, then re-apply Argo Events sensor manifests (or restart event-source pod).

=== GitHub Token (Workflow Dispatch)

**Path:** `/platform/github-token`

**Format:**
[source,json]
----
{
  "token": "ghp_..."
}
----

**Scope:** `workflow:write` (allows `workflow_dispatch` trigger from Argo Workflows)

**Usage:** Argo Workflows pod (`platform-argo-workflow-runner` SA) dispatches GitHub Actions workflows.

**Rotation:** Every 90 days. Manual rotation via GitHub token regeneration, then update secret in AWS Secrets Manager.

**Storage:** Also stored as GitHub org-level Actions secret `GH_TOKEN` (injected at job runtime). Keep both in sync.

=== Slack Webhook URL

**Path:** `/platform/slack-webhook`

**Format:**
[source,json]
----
{
  "url": "https://hooks.slack.com/services/..."
}
----

**Usage:** Slack notifications from Argo Workflows and Argo Events.

**Channels:** `#platform-alerts`, `#cluster-events`

**Rotation:** On Slack side: regenerate incoming webhook URL, update secret in AWS Secrets Manager.

== External Secrets Operator (ESO)

=== ClusterSecretStore Pattern

Each tenant or management cluster has one `ClusterSecretStore` that binds to an IRSA role.

**Example — tenant cluster:**

[source,yaml]
----
apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: aws-secretsmanager
  namespace: external-secrets
spec:
  provider:
    aws:
      service: SecretsManager
      region: eu-west-1
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets
            namespace: external-secrets
----

**ServiceAccount annotation** (in `external-secrets` namespace):

[source,yaml]
----
apiVersion: v1
kind: ServiceAccount
metadata:
  name: external-secrets
  namespace: external-secrets
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::<account>:role/<tenant-id>-external-secrets
----

**Namespace:** `external-secrets` (system app)

=== ExternalSecret Resource

Tenants create `ExternalSecret` resources in their namespace to sync secrets from AWS Secrets Manager.

**Example:**

[source,yaml]
----
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: database-credentials
  namespace: acme-corp
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secretsmanager
    kind: ClusterSecretStore
  target:
    name: database-secret           # Kubernetes Secret name
    creationPolicy: Owner
    deletionPolicy: Delete
  data:
    - secretKey: username
      remoteRef:
        key: /acme-corp/database/postgres-password
        property: username
        version: AWSCURRENT
    - secretKey: password
      remoteRef:
        key: /acme-corp/database/postgres-password
        property: password
        version: AWSCURRENT
----

**Parameters:**
- `refreshInterval`: How often to sync (default: 1h); shorter intervals cost more API calls
- `version: AWSCURRENT`: Always pull the latest version (required for auto-rotation)
- `property`: Extract a JSON field from the secret (optional; if omitted, assumes secret value is a single string)

=== Sync Flow

[source,mermaid]
----
sequenceDiagram
    participant ESO as ESO Controller
    participant ASM as AWS Secrets Manager
    participant K8s as Kubernetes API
    participant Pod as Pod

    loop refreshInterval (1h)
        ESO->>ESO: Watch ExternalSecret CR
        ESO->>ASM: Authenticate via IRSA<br/>(ServiceAccount annotation)
        ASM->>ASM: Validate JWT sub condition<br/>(namespace:serviceaccount)
        ESO->>ASM: GetSecretValue /path/to/secret
        ASM->>ESO: Return secret (AWSCURRENT version)
        ESO->>K8s: Create/Update Kubernetes Secret
        K8s->>K8s: Validate & store
    end

    Pod->>K8s: Mount Secret as env var or volume
    K8s->>Pod: Provide secret data
----

**Rotation behavior:**
- **Volume mounts:** Updated in-place; pods see new value without restart
- **Environment variables:** Require pod restart; use `reloader` or `rotate-secrets` workflow

== Creating & Rotating Secrets

=== Step 1 — Store in AWS Secrets Manager

[source,bash]
----
# Create a new secret
aws secretsmanager create-secret \
  --name /acme-corp/database/postgres-password \
  --secret-string '{"username":"dbuser","password":"p@ssw0rd"}' \
  --region eu-west-1

# Update an existing secret (creates new version AWSCURRENT)
aws secretsmanager update-secret \
  --secret-id /acme-corp/database/postgres-password \
  --secret-string '{"username":"dbuser","password":"newp@ssw0rd"}' \
  --region eu-west-1
----

=== Step 2 — Create ExternalSecret Manifest

Add to tenant's ArgoCD application:

[source,yaml]
----
# tenants/acme-corp/argocd/secrets.yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
  namespace: acme-corp
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secretsmanager
    kind: ClusterSecretStore
  target:
    name: db-credentials
    creationPolicy: Owner
  data:
    - secretKey: username
      remoteRef:
        key: /acme-corp/database/postgres-password
        property: username
    - secretKey: password
      remoteRef:
        key: /acme-corp/database/postgres-password
        property: password
----

=== Step 3 — Pod Mounts the Secret

**Environment variables:**

[source,yaml]
----
env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: db-credentials
        key: password
----

**Volume mount:**

[source,yaml]
----
volumeMounts:
  - name: db-creds
    mountPath: /etc/db-secrets
    readOnly: true

volumes:
  - name: db-creds
    secret:
      secretName: db-credentials
      defaultMode: 0400
----

=== Rotating a Secret

**Automatic rotation** (for secrets with Lambda rotators):

1. AWS Secrets Manager automatically updates the secret value (via Lambda)
2. New version is marked `AWSCURRENT`
3. ESO picks up the change on next refresh (default: 1h)
4. Kubernetes Secret is updated

**For environment variable injection,** the pod must restart to pick up the new value:

[source,bash]
----
# Trigger a rolling restart of deployments using the secret
kubectl rollout restart deployment/my-app -n acme-corp
----

Or use the `rotate-secrets` Argo Workflow (runs monthly, handles this automatically for all deployments).

**Force immediate ESO sync:**

[source,bash]
----
kubectl annotate externalsecret db-credentials \
  -n acme-corp \
  force-sync="$(date +%s)" \
  --overwrite
----

== GitHub Actions Secrets

GitHub Actions has **two separate** secret storage mechanisms:

=== GitHub Organization Secrets (for GHA jobs)

Stored at the GitHub org level; injected as environment variables in workflow steps.

**Used by:** GitHub Actions workflows (before any AWS auth)

| Secret | Purpose |
|---|---|
| `TERRAFORM_ROLE_ARN` | ARN of `platform-terraform-execution` role to assume via OIDC |
| `TERRAFORM_STATE_BUCKET` | S3 bucket for Terraform state |
| `TERRAFORM_LOCK_TABLE` | DynamoDB table for state locking |
| `SLACK_WEBHOOK_URL` | Slack incoming webhook (can be same as `/platform/slack-webhook` or different) |

=== AWS Secrets Manager Secrets (for in-cluster components)

Accessed from within the cluster via ESO.

| Secret | Path | Consumer | Rotation |
|---|---|---|---|
| GitHub PAT | `/platform/github-token` | Argo Workflows pod | Every 90 days |
| Slack webhook | `/platform/slack-webhook` | Argo Workflows + Events pods | On change |
| Webhook HMAC | `/platform/webhook-hmac-secret` | Argo Events | Manual |

**Why the split?**

- GitHub Actions runs *before* the cluster exists (bootstrap phase)
- Can't fetch from AWS Secrets Manager until after AWS credentials are configured
- GHA secrets are injected directly as environment variables
- In-cluster secrets are synced via ESO (more flexible, supports rotation)

== Pre-commit Secret Detection

All commits are scanned by `detect-secrets` pre-commit hook:

[source,bash]
----
# Install hook
pre-commit install

# Scan for secrets
pre-commit run detect-secrets --all-files
----

**Patterns detected:**
- AWS access keys / secret keys
- Private RSA/EC2 keys
- High-entropy strings (API tokens, passwords)
- Common secret names (`password`, `api_key`, `secret`, etc.)

**If a secret is accidentally committed:**

1. `git reset HEAD` — unstage the file
2. Edit to remove the secret
3. `git commit --amend` — amend the commit
4. **Rotate the secret immediately** in AWS Secrets Manager
5. Force-push to the branch (only if not yet merged to main)

== Emergency Access

=== Break-Glass Role

Used only when ESO is unavailable or urgent debugging is needed.

**Role:** `platform-break-glass` (requires MFA)

**Permissions:** Read-only access to all AWS Secrets Manager secrets

**Access:**

[source,bash]
----
# Authenticate with MFA
aws sts assume-role \
  --role-arn arn:aws:iam::<account>:role/platform-break-glass \
  --role-session-name emergency \
  --duration-seconds 3600 \
  --serial-number arn:aws:iam::<account>:mfa/your-device \
  --token-code 123456

# Set environment from returned credentials, then read secret
aws secretsmanager get-secret-value --secret-id /acme-corp/database/postgres-password
----

**Important:** Do NOT copy secrets to local files or shell history. Access only in-memory and via pipes.

**Logging:** All break-glass access is logged to CloudTrail and triggers an alert to `#platform-security`.

== Secret Scanning

=== GitHub Secret Scanning

GitHub's native secret scanning is enabled org-wide:
- Scans all pushed code for known secret patterns (AWS keys, tokens, etc.)
- Alerts the repository owner
- Can revoke exposed credentials

=== Weekly Historical Scan

A scheduled CI job runs `truffleHop` across full git history weekly:

[source,bash]
----
# Manual run
truffleHop --regex /path/to/patterns.json
----

**If exposed secrets are detected:**

1. Rotate the secret immediately in AWS Secrets Manager
2. Force-push to remove the secret from history (if not yet merged to main)
3. Post-incident review to determine exposure scope

== Troubleshooting

=== ExternalSecret Not Syncing

[source,bash]
----
kubectl describe externalsecret db-credentials -n acme-corp
# Check .status.conditions for error message
----

**Common issues:**
- **`AuthorizationFailed`** — IRSA role missing `secretsmanager:GetSecretValue` permission on the path
- **`SecretNotFound`** — Secret path is wrong or doesn't exist in Secrets Manager
- **`RefreshInterval not reached`** — Normal, wait for next refresh cycle (default 1h)

**Verify IAM role:**

[source,bash]
----
aws secretsmanager get-secret-value --secret-id /acme-corp/database/postgres-password
# Should succeed if ESO's IRSA role has the permission
----

=== Pod Not Receiving Updated Secret

For **volume mounts** (file-based): restart the pod, it will pick up the latest Kubernetes Secret value.

For **environment variables**: you must trigger a pod restart:

[source,bash]
----
kubectl rollout restart deployment/my-app -n acme-corp
----

=== Secret Changed But Pods Not Updated

Force ESO to re-sync immediately:

[source,bash]
----
kubectl annotate externalsecret db-credentials \
  -n acme-corp \
  force-sync="$(date +%s)" \
  --overwrite

# Wait a few seconds for ESO to pick up the annotation
kubectl get secret db-credentials -n acme-corp -w
----

== Security Checklist

- [ ] No secrets committed to Git (pre-commit hook enabled)
- [ ] All secret access paths scoped to tenant prefix in IRSA role
- [ ] ExternalSecrets use `version: AWSCURRENT` (required for auto-rotation)
- [ ] Volume mounts preferred over env vars (rotate without pod restart)
- [ ] Break-glass role accessed only in emergencies (logged, audited, alerted)
- [ ] Secrets rotated on schedule (monthly for managed secrets, per-SLA for others)
- [ ] GitHub PAT has minimal scopes (`workflow:write` only, no repo access)
- [ ] Webhook HMAC updated after rotation
