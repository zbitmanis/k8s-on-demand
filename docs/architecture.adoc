== System Purpose

Multi-tenant Kubernetes-as-a-Service platform built on a single shared AWS EKS cluster.
Each tenant receives a dedicated namespace with enforced isolation via:

* NetworkPolicy (network-level access control)
* ResourceQuota (compute limits)
* RBAC (authorization)
* Gatekeeper (admission policies)
* ESO (secrets delivery with path-based scoping)

Cluster is fully lifecycle-managed: provisioned on demand when work begins, destroyed when idle (zero cost).
Cold start ~20-30 minutes; no hibernation or scale-to-zero needed.

=== System Node Group Requirements

All system components **must** specify both `tolerations` and `nodeSelector`:

[source, yaml]
----
tolerations:
  - key: node-role
    value: system
    effect: NoSchedule

nodeSelector:
  node-role: system
----

**Reason:** Workload node group starts at `desired_size=0`. Without these constraints, system pods remain `Pending` until a workload node is scheduled by CA, breaking the separation and preventing cluster bootstrap.

== Tenant Isolation Model

[cols="1,2,3"]
|===
| Boundary | Mechanism | Enforcement & Scope

| **Network Isolation**
| NetworkPolicy default-deny per namespace + explicit allow rules
a|
* Calico or Cilium CNI enforces rules
* Every tenant namespace has `deny-all-ingress` policy by default
* Only allows traffic from: same namespace, ingress controller, system namespaces
* Egress allowed by default (can be restricted per tier)
* Gatekeeper blocks `allowAllIngress: true` policies

| **Compute Isolation**
| ResourceQuota per namespace + LimitRange per pod
a|
* Kubernetes API admission enforces quota
* Each tier (dev/staging/prod) gets configured quota in `config.yaml`
* CPU, memory, storage limits prevent pod starvation
* LimitRange sets minimum/maximum per container

| **RBAC / Access Control**
| Namespace-scoped service accounts, roles; cluster-roles forbidden
a|
* Tenant service accounts **cannot** access other namespaces
* Gatekeeper blocks any `ClusterRoleBinding` or `ClusterRole` creation by tenant
* Tenant **cannot** read/list other tenant secrets, configmaps, pods
* ArgoCD Project scoped per tenant namespace (isolation via RBAC)

| **Secrets Management**
| AWS Secrets Manager + ESO ClusterSecretStore scoped to `/<tenant-id>/*` prefix
a|
* ESO service account bound to dedicated IAM role
* Role policy restricts reads to `arn:aws:secretsmanager:*:*:secret:/<tenant-id>/*`
* ExternalSecret per namespace watches `/<tenant-id>/*` secrets
* Kubernetes Secret created only in tenant namespace
* Tenant **cannot** access other tenants' secrets (IAM policy + namespace isolation)

| **Pod Security**
| Gatekeeper constraints: no privileged pods, no host network, no host mounts, no system taint toleration
a|
* Gatekeeper admission controller enforces at pod creation
* Blocked: `securityContext.privileged=true`, `hostNetwork=true`, `hostPID=true`
* Blocked: volume types requiring host access (hostPath, etc.)
* Blocked: toleration for `node-role=system` (prevents tenant pods on system nodes)
* Enforcement mode: `warn` (audit only, pods not rejected yet)
* Requires full compliance audit before switching to `deny`

| **Storage**
| Namespace-scoped PVCs; host mounts forbidden
a|
* Kubernetes API restricts PVC creation to owning namespace
* Gatekeeper blocks hostPath and other privileged volume types
* Storage class (gp3) is shared but PVCs are namespace-isolated
* Backups tagged by tenant for billing/isolation

|===

== Tenant Onboarding Flow

[plantuml]
----
@startuml
autonumber

actor User as user
participant "GitHub / API" as api
participant "Argo Events" as ae
participant "Argo Workflows" as wf
participant "Kubernetes API" as k8s
participant "Crossplane" as cxp
participant "AWS API" as aws
participant "ArgoCD" as argocd

user ->> api: POST /onboard-tenant\n{tenant_id, tier, email}
api ->> ae: HTTP webhook received\n(EventSource listener)
ae ->> ae: Validate event\n(schema, fields)
ae ->> wf: Trigger Argo Workflow\n(Sensor → WorkflowTrigger)
ae -->> api: Return 202 Accepted

wf ->> wf: Validate tenant config\n(quota tier exists, naming)
wf ->> k8s: Create namespace\n(labeled: tenant-id=${tenant-id})
wf ->> k8s: Apply NetworkPolicy\n(default-deny + allow rules)
wf ->> k8s: Create ResourceQuota\n(cpu, memory, pods from tier config)
wf ->> k8s: Apply LimitRange\n(min/max per container)
wf ->> k8s: Create ServiceAccount\n(tenant-admin for RBAC)
wf ->> k8s: Create namespace Role\n(admin, read workloads)
wf ->> k8s: Create ClusterSecretStore binding\n(ESO scoped to /${tenant-id}/*)
wf ->> k8s: Create ArgoCD Project\n(namespace-scoped, RBAC)

wf ->> cxp: Create Crossplane Claims\n(XRCs: S3, IAM, RDS, SQS)
cxp ->> aws: Fulfill Claims\n(AWS resources created)
aws -->> cxp: Resources provisioned\n(S3 bucket, IAM role, RDS instance)
cxp -->> k8s: XRC status Ready

wf ->> k8s: Verify namespace ready\n(all conditions green)
wf -->> user: Namespace provisioned\n(~2–3 min total)

argocd ->> k8s: ApplicationSet watches tenants/\n(Git directory per tenant)
argocd ->> k8s: Sync tenant apps\n(Helm charts, manifests)
k8s -->> user: Tenant workloads deployed

@enduml
----

**Timeline:**
1. Webhook received → 1 sec
2. Namespace + RBAC + quotas + secrets binding → ~30 sec
3. Crossplane Claims fulfilled (AWS resources) → ~60–90 sec
4. ArgoCD syncs tenant apps → ~30–60 sec
5. **Total: ~2–3 minutes**

== Design Rationale

=== Single Shared Cluster (vs Dedicated Clusters per Tenant)

**Chosen:** Single shared cluster with namespace isolation

**Trade-offs:**

* ✓ **Massive cost reduction** — single control plane, single monitoring stack
* ✓ **Faster onboarding** — tenant creation in ~2–3 minutes (namespace + RBAC, no IaC)
* ✓ **Simpler operations** — one cluster to patch, monitor, update
* ✗ **Smaller blast radius** — one tenant's misconfiguration can affect others (mitigated by Gatekeeper, NetworkPolicy, quotas)
* ✗ **Cluster upgrade affects all** — no tenant-specific control plane versions (acceptable for dev/sandbox)
* ✗ **Resource contention** — requires careful quota tuning per tier (defined in `config.yaml`)

=== Crossplane Compositions for Day 1 AWS Resources (vs Terraform)

**Chosen:** Crossplane Compositions (GitOps)

**Rationale:**

* ✓ **Kubernetes-native API** — tenants request resources via Claims (XRCs), no Terraform state
* ✓ **GitOps delivery** — tenant config in Git, ArgoCD syncs Claims, Crossplane fulfills
* ✓ **Stateless tenant onboarding** — claims are declarative; reapplying is idempotent
* ✓ **No per-tenant Terraform state** — single Crossplane ProviderConfig manages all tenants
* ✗ **Crossplane complexity** — requires learning XRDs, Compositions, Claims model
* ✗ **Debugging** — Crossplane status conditions can be harder to read than `terraform state`

**Key rule:** Never add per-tenant AWS resources to Terraform. Terraform is Day 0 only (cluster infrastructure); all Day 1 tenant resources go through Crossplane Compositions.

=== Argo Workflows for Namespace Provisioning (vs Static Manifests)

**Chosen:** Argo Workflows (orchestrated, multi-step)

**Rationale:**

* ✓ **Stateful orchestration** — namespace → quota → RBAC → ESO → ArgoCD Project (sequential, validated)
* ✓ **Error handling** — workflow fails on validation errors, logs audit trail
* ✓ **Audit & retry** — each step is logged, step can be retried without re-creating namespace
* ✓ **Event-driven** — integrates with Argo Events (webhook → sensor → workflow trigger)
* ✗ **Complexity** — more moving parts than static manifests
* ✗ **Debugging** — requires understanding Argo Workflow DAGs and templating

=== Single Prometheus (vs Per-Tenant)

**Chosen:** Single Prometheus instance, label by namespace

**Rationale:**

* ✓ **Cost & operations** — one scrape loop, one TSDB, one backup
* ✓ **Cross-tenant queries** — Platform team can see all metrics (useful for platform improvements)
* ✓ **Label isolation** — Grafana/Thanos filter by `namespace` label; tenant cannot see others' metrics
* ✗ **Noisy metrics** — single instance sees all workloads (mitigated by reasonable cardinality limits)

=== ApplicationSet Pattern (vs Per-Tenant Applications)

**Chosen:** ApplicationSet with tenant generator (per-directory discovery)

**Rationale:**

* ✓ **DRY** — one ApplicationSet template, dynamically creates Applications per tenant
* ✓ **Auto-discovery** — add `tenants/<tenant-id>/` directory, ApplicationSet creates Application
* ✓ **Namespace isolation** — each Application scoped to tenant namespace
* ✗ **Harder to debug** — template variables can be confusing

=== Node Group Scaling Split

**Chosen:** Lambda + EventBridge for system nodes; Cluster Autoscaler for workload nodes

**Rationale:**

* **System nodes (0–3):** Lambda/EventBridge enables explicit suspend/resume on schedule (cost saving)
  * Cannot use CA because system pods run 24/7 (ArgoCD, Prometheus never idle)
  * CA would never scale them down
* **Workload nodes (0–20):** CA scales reactively based on pending pod demand
  * Cannot use fixed schedule because tenant workloads arrive unpredictably
  * CA uses least-waste bin packing to minimize node count
  * Scale-from-zero enabled via ASG resource hints

**Important:** System node group ASG is **not tagged** for CA auto-discovery (`k8s.io/cluster-autoscaler/enabled` tag absent). Only workload group has CA tags. Prevents CA from touching system nodes.

=== Cluster Autoscaler Configuration

See `src/applications/cluster-autoscaler/` Helm values for full config.

**Key settings:**

* `expander: least-waste` — chooses node type that minimizes wasted CPU/memory
* `scale-down-delay: 5m` — aggressive drain (dev/sandbox context tolerates disruption)
* `scale-from-zero: true` — enabled via ASG resource hints (`cpu=2, memory=4Gi`)
  * Without hints, CA cannot estimate node capacity and will not scale from 0
* `skip-nodes-with-system-pods: false` — safe because system pods run on system nodes (separate ASG)

**IRSA role:** `${cluster_name}-cluster-autoscaler`
* Policy: `SetDesiredCapacity`, `TerminateInstanceInAutoScalingGroup` on workload group ASG only
* Condition: ASG tagged with `k8s.io/cluster-autoscaler/enabled=true`

=== Cluster Suspend / Resume

**Soft path (preferred for development):**
* Argo Workflow triggered by webhook or manual dispatch
* Gracefully scales node groups to 0
* Cluster remains provisioned; pods evicted cleanly

**Hard path (cost optimization):**
* EventBridge cron schedule (5pm daily)
* Lambda scales node groups to 0
* Pods interrupted (acceptable for dev/sandbox)

**Resumption:**
* EventBridge cron (9am daily) or manual
* Lambda scales system ASG up
* ArgoCD re-syncs all apps
* Workload ASG scales on demand
* Total time: ~5–7 min

=== Bootstrap Buckets (Survive Cluster Destroy)

**Provisioned in:** `src/terraform/bootstrap/` (separate Terraform state, not in main cluster state)

**Buckets:**
* `k8s-od-thanos-metrics` — long-term metrics (survives cluster destroy)
* `k8s-od-argo-artifacts` — workflow artifacts (survives cluster destroy)

**Why separate:**
* Cluster can be destroyed and re-provisioned without losing historical data
* Terraform state for bootstrap is distinct from cluster state
* Allows cluster reprovision using same S3 buckets

=== On-Demand Cluster Lifecycle

**Cold start:** ~20–30 minutes from `terraform apply` to cluster ready
* Terraform provisioning: ~15–18 min
* System apps bootstrap: ~3–5 min
* Total: ~20–25 min

**No hibernation or scale-to-zero needed:** Cluster is fully destroyed when idle (zero control plane cost).
Reprovision on next work session.

**State recovery:**
* Git: all IaC and config (complete)
* AWS Secrets Manager: all tenant secrets (not deleted)
* S3: Thanos metrics, Argo artifacts (not deleted)
* Terraform state: S3 backend (versioned)

== Key Constraints & Rules

1. **Never `terraform apply` locally** — all applies must go through GitHub Actions with OIDC
2. **Terraform is Day 0 only** — no per-tenant resources in Terraform
3. **Crossplane Compositions are the Day 1 API** — tenants request AWS resources via Claims (XRCs)
4. **ArgoCD is the source of truth** — no `kubectl apply` directly; all state in Git
5. **Gatekeeper always on** — never disable admission webhooks without platform lead approval
6. **IRSA only** — every service with AWS access gets its own dedicated role; no static credentials
7. **Tenant isolation is mandatory** — every PR touching tenant config requires isolation review
8. **System components must tolerate system taint** — all system workloads require `tolerations` and `nodeSelector`
9. **System node group is not CA-managed** — no CA tags on system ASG; Lambda/EventBridge control suspend/resume
10. **NetworkPolicy must default-deny** — explicit allow rules per namespace, no allow-all policies
